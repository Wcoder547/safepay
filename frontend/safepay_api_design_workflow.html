URL nesting decisions
GET /wallet/balance ← wallet belongs to logged-in user (JWT) GET /transactions/:id ← flat, not nested — transactions accessed directly GET /transactions/:id/fraud ← one level nest — fraud belongs to transaction GET /notifications ← user implicit via JWT, no nesting needed
We keep nesting shallow. User context comes from JWT token — so /wallet/balance, not /users/:id/wallet/balance. Flatter URLs are cleaner and easier to consume.

SafePay openapi.yaml — starter
openapi: 3.0.3
info:
  title: SafePay API
  description: Digital wallet with AI fraud detection
  version: 1.0.0

servers:
  - url: http://localhost:8000/v1
    description: Local dev
  - url: https://api.safepay.pk/v1
    description: Production

components:
  securitySchemes:
    BearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT

  schemas:
    Error:
      type: object
      properties:
        success: { type: boolean, example: false }
        error:
          type: object
          properties:
            code: { type: string }
            message: { type: string }
            field: { type: string }

    User:
      type: object
      properties:
        id: { type: string, format: uuid }
        full_name: { type: string }
        email: { type: string, format: email }
        phone: { type: string }
        role: { type: string, enum: [user, admin] }
        is_verified: { type: boolean }
        is_frozen: { type: boolean }
        created_at: { type: string, format: date-time }

    Wallet:
      type: object
      properties:
        id: { type: string, format: uuid }
        balance: { type: string, example: "15000.00" }
        currency: { type: string, example: PKR }

paths:
  /auth/register:
    post:
      tags: [Auth]
      summary: Register new account
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [full_name,email,phone,cnic,password,pin]
              properties:
                full_name: { type: string }
                email: { type: string, format: email }
                phone: { type: string, example: "03001234567" }
                cnic: { type: string, example: "42201-1234567-1" }
                password: { type: string, minLength: 8 }
                pin: { type: string, pattern: "^[0-9]{4}$" }
      responses:
        "201": { description: Account created }
        "409": { description: Duplicate email/phone/CNIC }
        "422": { description: Validation error }
        "429": { description: Rate limit exceeded }
Once your YAML is written, paste it into editor.swagger.io — it instantly generates a live interactive API playground. Your React frontend team can use this to mock responses before backend is built.
